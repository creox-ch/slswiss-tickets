# Подключение нового сайта к /api/forms

Формы платформенных сайтов (chudina, atlasintegra, creox, frankenplatz …) живут
на своих доменах и шлют заявки в `POST /api/forms` этого репо. Новый сайт не
заработает, пока его origin не попал в белый список: route отвечает
`403 forbidden origin`.

## Что нужно от пользователя

Домены нового сайта. Обычно их три-четыре:
`https://<домен>`, `https://www.<домен>`, `https://<проект>.vercel.app`.
Если пользователь назвал только один — спроси про остальные: сайт на Vercel
почти всегда доступен и по vercel.app-адресу, а форма с www-версии домена
отправляет origin с `www.`, и он тоже должен быть в списке.

Origin — это `схема://хост[:порт]`, без пути и без завершающего слэша.
`https://creox.ch/kontakt` — не origin.

## Шаги

### 1. Добавить домены в DEFAULT_ORIGINS

В [`lib/forms.js`](lib/forms.js) — массив `DEFAULT_ORIGINS`. Вставляй новые
домены группой, рядом с соседними сайтами, сохраняя порядок «голый домен →
www → vercel.app». `http://localhost:3000` оставляй последним.

### 2. Проверить env FORMS_ALLOWED_ORIGINS на проде

**Главные грабли этого репо.** `allowedOrigins()` устроен так, что env
`FORMS_ALLOWED_ORIGINS` заменяет `DEFAULT_ORIGINS` **целиком**, а не дополняет.
Если на Vercel эта переменная задана, правка кода ничего не изменит — новый
сайт по-прежнему получит 403.

Сам ты в Vercel не лезешь. Скажи пользователю проверить:
Vercel → проект `slswiss-tickets` → Settings → Environment Variables →
`FORMS_ALLOWED_ORIGINS`.

- Переменной нет → всё в порядке, работает `DEFAULT_ORIGINS`.
- Переменная есть → в неё нужно вручную дописать новые домены через запятую,
  иначе деплой не поможет. Предупреди об этом **до** коммита, а не после.

### 3. Дописать тесты

В [`tests/unit/forms.spec.js`](tests/unit/forms.spec.js), блок
`test.describe('allowedOrigins')`, первый тест «без env — дефолтные сайты
платформы»: добавь `expect(list).toContain('https://<новый домен>')` на каждый
добавленный домен.

Если у сайта своя форма со своей структурой полей (как блок «заявка с лендинга
creox»), добавь такой же describe-блок с `normalizeSubmission` под её payload:
проверь `source`, `form_key`, `kind`, нормализацию email, `consent` и пустой
honeypot. Тесты пишешь сам, пользователя об этом не спрашиваешь — в проекте так
принято.

### 4. Проверить

```powershell
node --check lib/forms.js
npx playwright test tests/unit/forms.spec.js
```

Локального node может не быть — тогда синтаксис и тесты проверит CI после
пуша (`.github/workflows/test.yml`). Не выдавай непроверенное за проверенное:
если тесты локально не гонялись, так и скажи и дождись зелёного CI.

### 5. Закоммитить

Git — из PowerShell, не из песочницы (в mount застревает `.git/index.lock`;
если он остался от прошлого раза — удали и повтори). Один коммит, сообщение в
стиле репо:

```
feat(forms): приём заявок с <сайта> (origins + тесты)
```

Покажи `git diff --stat` перед коммитом.

## Проверка после деплоя

Vercel деплоит `main` автоматически. Убедиться, что origin принят, можно
preflight-запросом — он не создаёт заявку в базе:

```powershell
curl.exe -i -X OPTIONS https://slswiss-tickets.vercel.app/api/forms -H "Origin: https://<новый домен>" -H "Access-Control-Request-Method: POST"
```

В ответе должен быть заголовок `Access-Control-Allow-Origin` с этим доменом.
Нет заголовка → origin не в списке: скорее всего сработала env из шага 2.

Живую отправку формы проверяет пользователь на самом сайте — заявка при этом
попадёт в `public.submissions` и уйдёт письмом, это реальная запись.
